## 1. Purpose

This process automates the intake and initial resolution of card disputes pulled from a work queue, validating that the underlying account is active and that the disputed transaction genuinely occurred, then either routing high-value disputes to a human case reviewer or automatically issuing a provisional credit (dispute amount plus a calculated fee refund) by posting an adjustment to the core banking system. It includes retry logic for posting failures and ensures every dispute item is ultimately completed or exceptioned on the queue, providing a controlled, auditable path from customer dispute claim to either automated remediation or manual review.

Evidence: Get Next Dispute (queue_read); Unpack Dispute Item (calculation); Account Active? (decision); Transaction Found? (decision); Amount Over Review Threshold? (decision); Post Adjustment (action); Mark Dispute Completed (queue_write)

## 6. Business rules

**BR-1.** A dispute can only proceed if the associated account is active; otherwise it is exceptioned as 'Account not active'.

Evidence: Account Active? (decision); Mark Exception - Inactive Account (queue_write)

**BR-2.** The dispute must correspond to an actual transaction matching both the transaction date and disputed amount; if no matching transaction is found, the item is exceptioned as 'No matching transaction'.

Evidence: Matches Disputed Transaction? (decision); Transaction Found? (decision); Mark Exception - No Matching Transaction (queue_write)

**BR-3.** Disputes with an amount exceeding the configured review threshold are not auto-processed; a manual review case is created and the item is deferred rather than resolved automatically.

Evidence: Amount Over Review Threshold? (decision); Create Review Case (action); Defer For Review (queue_write)

**BR-4.** For disputes within the threshold, a fee refund is calculated as the dispute amount times a fee rate, and the provisional credit posted equals dispute amount plus fee refund.

Evidence: Compute Fee Refund (calculation); Compute Provisional Credit (calculation)

**BR-5.** If posting the adjustment fails, the process retries up to a maximum number of attempts before giving up and exceptioning the item as 'Posting failed after retries'.

Evidence: Retry Limit Reached? (decision); Increment Retry (calculation); Retry Posting (resume); Give Up (resume); Mark Exception - Posting Failed (queue_write)

## 7. Current exception handling

| When | Current handling | Evidence |
|---|---|---|
| Account associated with the dispute is not active | Item is marked as an exception on the queue with reason 'Account not active' and the process moves to the next item | Account Active? (decision); Mark Exception - Inactive Account (queue_write) |
| No transaction on the account matches the disputed date and amount | Item is marked as an exception with reason 'No matching transaction' | Transaction Found? (decision); Mark Exception - No Matching Transaction (queue_write) |
| Dispute amount exceeds the review threshold | A manual review case is created and the queue item is marked as an exception with reason 'Deferred for manual review' | Amount Over Review Threshold? (decision); Create Review Case (action); Defer For Review (queue_write) |

## 9. Human touchpoints

- High-value disputes above the review threshold are routed to a human reviewer via a case created in the case management system, and the automated processing is deferred pending that manual decision
  Evidence: Amount Over Review Threshold? (decision); Create Review Case (action); Defer For Review (queue_write)
- Disputes exceptioned for inactive account, no matching transaction, or repeated posting failures are left on the queue in an exception state, implying a human must investigate and resolve them
  Evidence: Mark Exception - Inactive Account (queue_write); Mark Exception - No Matching Transaction (queue_write); Mark Exception - Posting Failed (queue_write)
